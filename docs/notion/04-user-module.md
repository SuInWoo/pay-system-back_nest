- **역할**: (예: 회원/비회원 사용자 관리, 인증/인가와 주문·결제 연동)

## 2. 아키텍처 · 설계

- **모듈 구조**
  - (예: `UserModule`, `UserService`, `UserController`, `UserRepository` 등)
- **핵심 엔티티/테이블**
  - (예: `users`, `user_profiles` 등)
- **보안·개인정보 관점 고려 사항**
  - 비밀번호 저장 방식, 토큰 관리 (예: 해시 알고리즘, 만료·재발급 정책)
  - 마스킹/로그 정책: 이메일·전화번호 등은 필요 시 일부만 로그에 남기고, 민감한 정보는 절대 직접 로그에 남기지 않기
  - 민감 정보와 일반 정보의 분리: 인증·보안 관점에서 꼭 필요한 정보와 그렇지 않은 정보를 테이블/컬럼 레벨에서 구분

---

## 3. API 설계

- **주요 API 목록**
  - `POST /users` : 회원 가입
  - `POST /auth/login` : 로그인
  - `GET /me` : 내 정보 조회
  - (실제 구현에 맞게 수정/추가)
- **주문/결제와의 연동 포인트**
  - 유저 ID 기반으로 주문/결제 히스토리 조회
  - 비회원 주문/결제 시 전략(있다면)
  - 관리자/운영자 권한을 통해 특정 유저의 주문/결제 내역을 조회할 때, 개인정보 노출 범위를 어디까지 허용할지 정책화

## 6. 개인 회고 · 느낀 점

- **이번 챕터에서 느낀 점**
  - 유저 도메인이 주문·결제와 강하게 엮여 있어서, “유저 ID”가 거의 모든 로그·추적의 출발점이 된다는 걸 다시 체감했다.
- **배운 것**
  - 인증/인가를 “일단 로그인만 되게” 넣어두면, 나중에 권한·운영·감사 요구사항이 생겼을 때 거의 처음부터 다시 설계해야 한다는 교훈.
- **다음 챕터(k6·성능 테스트)에서 더 신경 쓸 것**
  - 유저 수/트래픽 스케일을 가정하고 k6 시나리오를 설계해, 인증/인가·세션·토큰 검증이 병목이 되지 않도록 미리 확인하기.
