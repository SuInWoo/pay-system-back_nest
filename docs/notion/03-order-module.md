## 1. 주문(OMS) 모듈 개요

- **역할**: (예: 상품 주문 생성부터 상태 변경, 결제 연동까지 전체 흐름 관리)
- **주요 책임**
  - 주문 생성/변경/취소
  - 결제 상태와의 동기화
  - 재고/배송(향후 확장)과의 연동 지점 마련
- **주요 연관 도메인**
  - 결제, 유저, 상품/재고(향후)

---

## 2. 아키텍처 · 설계

- **모듈 구조**
  - (예: `OrderModule`, `OrderService`, `OrderController`, `OrderRepository` 등)
- **핵심 엔티티/테이블**
  - (예: `orders`, `order_items` 등)
- **주문 상태 플로우**
  - (예: `CREATED → PAID → PREPARING → SHIPPED → COMPLETED / CANCELED` 등)
- **중요 설계 결정**
  - 주문과 결제의 경계: 주문은 “무엇을 얼마에 사고자 했다”까지, 결제는 “실제 돈이 어떻게 움직였는가”를 책임지는 것으로 구분한다.
  - 현재 단계에서는 구현 복잡도를 줄이기 위해 주문–결제를 동기 API 호출로 연동하고, 나중에 필요 시 이벤트 기반으로 확장할 수 있게 내부 인터페이스를 정리해 둔다.
  - **개인정보(PII) 암호화**: 주문자명, 주소, 연락처, 이메일은 AES-256-GCM으로 DB 저장 시 암호화. `PII_ENCRYPTION_KEY` 환경변수(32바이트 base64) 필수.

---

## 3. API 설계

- **주요 API 목록**
  - `POST /orders` : 주문 생성
  - `GET /orders/{id}` : 주문 조회
  - `PATCH /orders/{id}/cancel` : 주문 취소
  - (실제 구현에 맞게 수정/추가)
- **주문–결제 연동 포인트**
  - 주문 생성 후 결제 요청 트리거
  - 결제 결과에 따른 주문 상태 변경
  - 결제 실패/타임아웃 시 주문 상태를 어떻게 둘지(예: `CREATED` 유지 vs `FAILED` 상태 도입)를 문서·테스트로 먼저 정의하고 구현
