## 2. 아키텍처 · 설계

- **모듈 구조**
  - (예: `PaymentModule`, `PaymentService`, `PaymentController`, `PaymentRepository` 등)
- **핵심 엔티티/테이블**
  - (예: `payments`, `payment_attempts` 등 ERD 관점에서 요약)
- **상태 플로우**
  - (예: `REQUESTED → AUTHORIZED → CAPTURED → CANCELED / FAILED` 등 상태 전이 정의)
- **중요 설계 결정**
  - 초기에는 단건 결제 플로우부터 안정화하고, 부분 취소/부분 환불·정산은 이후 단계로 미루기로 했다.
  - 외부 PG 의존성을 인터페이스로 추상화해 테스트/교체를 쉽게 하고, 주문 모듈에서는 `PaymentService`의 도메인 메서드만 사용하도록 경계를 그었다.
  - 결제 상태 전이는 주문 상태와의 정합성을 유지하도록 정의하고, 실패/타임아웃 케이스를 별도의 도메인 에러로 표현한다.

## 3. API 설계

- **주요 API 목록**
  - `POST /payments` : 결제 요청 생성
  - `POST /payments/{id}/confirm` : 결제 승인
  - `POST /payments/{id}/cancel` : 결제 취소
  - (실제 구현에 맞게 수정)
- **요청/응답에서 중요하게 본 포인트**
  - idempotency, 중복 결제 방지 (예: 클라이언트/주문 단위 idempotency key 사용)
  - 에러 코드/메시지 정책: 외부 PG 에러를 내부 도메인 에러코드(예: `PAYMENT_GATEWAY_ERROR`)로 감싸고, 로그에는 `orderId`, `userId`, `requestId`를 함께 남긴다.

## 6. 개인 회고 · 느낀 점

- **이번 챕터에서 느낀 점**
  - 결제 모듈 하나만 보더라도 주문/유저/정산과 강하게 연결되어 있어, “어디까지를 결제 책임으로 둘지”를 명확히 하지 않으면 금방 복잡해진다는 걸 느꼈다.
- **배운 것**

- **다음 챕터(주문 모듈)에서 더 신경 쓸 것**
  - 주문과 결제의 경계를 더 명확히 잡고, “결제 실패/취소가 주문에 어떻게 반영되는지”를 문서·테스트·로그 기준으로 정리해 두기.
