## 1. k6 도입 배경

- **왜 k6를 선택했는가**
  - 스크립트 기반으로 Git에 함께 버전 관리가 가능해, “어떤 부하 조건에서 어떤 결과가 나왔는지”를 기록으로 남길 수 있다.
  - CLI로 쉽게 실행할 수 있고, 추후 Grafana/Prometheus 등과 연동해 대시보드를 구성하기에도 적합하다.
- **목표**
  - 결제/주문/유저 핵심 플로우에 대한 성능 기준 설정
  - 병목 지점 찾기, 현실적인 트래픽 가정 테스트

---

## 2. 시나리오 설계

- **테스트 대상 플로우**
  - 주문 생성 → 결제 요청 → 결제 승인
  - 유저 가입/로그인 → 주문/결제
- **부하 패턴**
  - 점진적 증가(램프업)
  - 스파이크/버스트 테스트(예정)
- **성능 지표**
  - 응답 시간(p95, p99)
  - 에러율
  - TPS/요청 수
  - 특정 API에만 집중되는 병목 여부(예: 결제 승인, 로그인 등)

---

## 3. k6 스크립트 구조

- **폴더/파일 구조**
  - (예: `testing/` 또는 `docs/testing/` 아래에 k6 스크립트 위치)
- **스크립트 구성 요소**
  - 설정 옵션(VUs, duration 등)
  - 시나리오 함수(주문/결제/유저별)
  - 공통 유틸(토큰 발급, 공통 헤더 등)
  - 결과 요약을 로그/리포트 형태로 남겨, Notion/문서에 붙여넣을 수 있도록 포맷 정리

---

## 4. 실행 방법 · 결과 해석

- **실행 명령 예시**
  - `k6 run ./testing/payment_scenario.js`
  - (실제 경로/파일명에 맞게 수정)
- **결과에서 보는 포인트**
  - 응답 시간 분포
  - 실패 요청 유형(타임아웃, 5xx 등)
  - 특정 API에만 집중되는 병목 여부

---

## 5. 앞으로의 디벨롭 계획

- **단계 1 – 기본 시나리오 완성**
  - 핵심 플로우에 대해 단일 환경 기준의 벤치마크 만들기
- **단계 2 – 환경별 비교**
  - 로컬/스테이징(있다면) 간 비교
  - 설정/캐시/DB 인덱스 조정 전후 비교
- **단계 3 – 운영 수준 목표 설정**
  - 최소/목표 성능 기준 정하기
  - 장애/스파이크 시나리오 추가

---

## 6. 개인 회고 · 느낀 점

- **이번 챕터에서 느낀 점**
  - 막연한 “빠르게”가 아니라, 숫자로 보는 성능(p95, 에러율 등)이 있어야만 진짜 대화를 할 수 있다는 걸 느꼈다.
- **배운 것**
  - k6 시나리오를 코드로 관리하면, 환경·설정·코드 변경 전후의 성능 차이를 실험처럼 반복 검증할 수 있다는 점.
- **다음에 더 시도해보고 싶은 것**
  - Grafana/Prometheus 등과 연동해서 대시보드까지 구축하고, 주문/결제·유저 플로우를 실시간으로 모니터링해 보는 것.
