# 회의록: AI 결제·OMS 프로젝트 중요 포인트 전체 회고

- **일자**: 2026-03-11
- **참석 팀**: 아키텍처·개발팀, DB·성능팀, 품질·에러·로그팀, 보안·운영팀, 문서·노션팀, 비즈니스·유지보수팀

---

## 1. 회의 목적

AI를 팀처럼 활용해 개발 중인 결제·OMS Nest 프로젝트에서, 각 역할이 실제로 중요하게 보고 있는 지점과 개발 시 신경 쓴 부분을 공유하고,
이를 바탕으로 Notion/`docs/notion/*` 문서를 더 구체적으로 채우기 위한 기준을 정한다.

---

## 2. 팀별 의견

### 아키텍처·개발팀 (시니어담당자 중심)

- 모듈 경계를 `결제 / 주문 / 유저`로 명확히 나누고, 의존 방향을 한 방향(예: `User → Order → Payment`)으로 유지하는 것을 최우선으로 봤다.
- ERD와 `schema.sql`을 먼저 설계한 뒤 Nest 모듈을 올리는 순서를 지키려고 했다.
- 서비스 레이어에서 비즈니스 규칙이 코드에 드러나도록 메서드 이름과 DTO를 신경 썼다.
- AI를 “자동완성 도구”가 아니라 “역할이 있는 시니어/리뷰어”처럼 사용하려고, 팀 역할 규칙을 적극적으로 적용했다.

간단 예시:

```ts
@Module({
  imports: [TypeOrmModule.forFeature([PaymentEntity])],
  controllers: [PaymentController],
  providers: [PaymentService],
  exports: [PaymentService], // 주문 모듈에서만 의존
})
export class PaymentModule {}
```

### DB·성능팀 (DB담당자 + 성능담당자)

- 주문/결제/유저 테이블을 분리하되, 조회 패턴(유저 기준, 주문 기준, 결제 상태 기준)에 맞는 인덱스를 설계하는 것을 중요하게 봤다.
- 금액/상태/시간 컬럼 타입과 제약조건을 엄격하게 두어 정합성을 우선시했다.
- k6 도입으로 “느낌상 빠름”이 아니라, 숫자로 성능을 관리하는 문화를 초기에 심어두고자 했다.
- 트랜잭션 구간을 짧게 유지하고, 외부 PG 호출이 중첩되지 않도록 서비스 설계를 검토했다.

간단 예시:

```ts
@Entity('payments')
@Index(['userId', 'createdAt'])
@Index(['orderId'])
export class PaymentEntity {
  @PrimaryGeneratedColumn('uuid')
  id: string;

  @Column()
  userId: string;

  @Column()
  orderId: string;
}
```

### 품질·에러·로그팀 (QA담당자 + 에러담당자 + 로그담당자)

- 각 모듈별로 “테스트 전략”을 문서(`docs/notion/0x-*.md`) 안에 명시하는 것을 중요하게 봤다.
- Happy path 뿐 아니라 “결제 실패 시 주문 상태는?”, “외부 PG 타임아웃 시 재시도/롤백 전략은?” 같은 시나리오를 우선 문서로 정의하려 했다.
- 에러 메시지를 “사용자 노출용”과 “서버 내부 로깅용”으로 구분해 설계해야 한다고 강조했다.
- 로그에는 `orderId`, `userId`, `requestId` 같은 식별자를 공통으로 남겨, 나중에 장애 분석 시 흐름을 따라가기 쉽게 하자는 의견을 냈다.

간단 예시:

```ts
throw new PaymentDomainException('PAYMENT_GATEWAY_ERROR', {
  orderId,
  userId,
  requestId,
});
```

### 보안·운영팀 (보안담당자 + 운영담당자)

- 유저 모듈에서 인증/인가를 “대충 로그인만 되게”가 아니라, 토큰 만료/재발급, 권한(Role) 분리까지 염두에 두고 설계해야 한다고 봤다.
- 로그/모니터링/헬스 체크 포인트를 어디에 둘지 미리 생각해두고, k6 결과와 함께 운영 관점에서 볼 수 있게 하자는 의견을 냈다.
- 환경별 설정(실서버/로컬) 분리를 고려해, `.env` 및 설정 모듈 구조를 깔끔히 가져가야 나중에 배포가 편하다고 강조했다.

### 문서·노션팀 (비즈니스문서담당자 + 노션리뷰담당자 + API명세담당자)

- `docs/notion/01~05` 챕터 구조를 실제 개발/회의 내용과 계속 동기화하는 것이 중요하다고 봤다.
- 각 챕터 마지막의 “개인 회고 · 느낀 점”을 적극적으로 채워서, 단순 결과뿐 아니라 고민의 히스토리를 남기자는 의견을 냈다.
- API 명세는 Swagger에만 두지 말고, 핵심 흐름(주문→결제→정산)은 Notion/문서에도 간단한 시퀀스와 함께 적어두기로 했다.

### 비즈니스·유지보수팀 (비즈니스담당자 + 유지보수담당자)

- 일정/리소스를 고려해, “실무에서 바로 쓸 수 있는 최소한의 기능”에 먼저 집중하자는 의견을 냈다.
- 돈이 엮인 데이터(금액, 정산, 취소/환불)는 보수적으로 설계하고, 문서로 정확히 정의해두어야 한다고 강조했다.
- 운영자가 DB를 직접 뒤지지 않고도 상황을 파악할 수 있게, 관리자 API/화면의 필요성을 초기부터 염두에 두기로 했다.

---

## 3. 결론

- 모듈 경계(`결제 / 주문 / 유저`)와 의존 방향을 계속 유지하면서, 도메인 복잡도는 “문서 → 코드” 순으로 확장해 나가기로 합의했다.
- k6를 활용한 성능 테스트를 정기적으로 실행하여, 주요 플로우(주문→결제)를 수치로 관리하기로 했다.
- 에러/로그 정책을 미리 정의하고, `orderId`, `userId`, `requestId` 등 공통 식별자를 로그에 남기는 규칙을 적용하기로 했다.
- Notion/`docs/notion/*` 문서의 각 챕터에, 이번 회의 내용을 반영해 설계/테스트/회고 섹션을 구체적으로 채우기로 했다.
- 돈/정산/환불 관련 기능은 보수적으로 단계별로 확장하고, 각 단계마다 비즈니스 규칙을 문서로 남기기로 했다.
